What is the Cloud?

Computing and software resources that are delivered on demand, as service. (A Walk in the Clouds. Cloud Computing, CDW-G Reference Guide., 3-5)

Cloud Service Models

PaaS Management: A Multi-Objective Problem

PaaS elastic management as a cornerstone of PaaS (Platform As A Service) has to deal with trade-offs between conflicting requirements such as cost , quality of service and security . Current cloud platforms usually focus on one dimensional optimization .

Conflicting Objectives in VM Management

How to place VM hosting applications belonging to many customers?

Motivation

Most of PaaS providers consider only one optimization axis, local optimization that can be provided by a cloud provider to a cloud customer is typically one of the major components of a platform as a service (PaaS) hosting.

Objectives

Our goal is to provide a supervision approach to reconfigure a cloud platform according to evolving customer optimization objectives to:

  1. Adjust customers resources provisioning and quality of service to the deployment costs.
  2. Reduce data centers energy consumption at the level of the cloud provider.

CMOOP Modeling

Models@run.time for PaaS abstraction:

  1. Provide high level abstraction for Clouds through a standard MOF-based metamodel
  2. Enable to reason about the system at design time
  3. Extend the reasoning to the system while it is in a running state
  4. Reevaluate continuously requirements satisfaction while the system is evolving.

CMOOP Modeling

Kevoree [1] as a platform to run our implementation:

  1. A physical machine → Kevoree Infrastructure Node
  2. A virtual machine → Kevoree Child Node
  3. A component → Software service

CMOOP Resolution

Problem Definition: “Given a dedicated set of virtual machines allocated by a PaaS provider, how to optimize software placement in VMs to reduce costs and to maintain compliance with SLA requirements".

⇒ Search-based approaches for multi-objective optimization problems: genetic algorithms.

CMOOP Problem Domain Inputs

We define CMOOP by the following triplet (I,F,CO) such that:

  1. I denotes a cloud infrastructure model.
  2. F is a vector of objective functions F(X) = ( f1(x),f2(x),...,fn(X)), that have to be minimized.
  3. A configuration co (individual) ∈ CO is obtained through a mapping function that maps the components (C) to (VMs): co = VM1(c1,c2,c3) denotes a configuration with a single virtual machine VM1 hosting 3 components c1,c2,c3.

Operators Description

Operators Description
AddComponent(c,A) Adds a component c in the the virtual machine A
RemoveComponent(c,A) Removes a component c in the virtual machine A
MoveComponent(c,A,B) Moves a component c from the virtual machine A to the virtual machine B

Fitness Functions Definition

  1. f1(x)= Cost(x): calculated based on Amazon pricing model (http://aws.amazon.com/ec2/pricing/)
  2. f2(x)=Security(x) defines the security level associated with a Component
  3. f3(x)=Completeness(x) provides a score of the number of software components that have been placed
  4. f4(x)=Overload(x) defines the gap between the required and the observed % of virtual CPU for each software element
  5. f5(x)=SLAPerformance(x) defines the required CPU for every component as stated in the SLA

CMOP Algorithm: NSGA II

Validation Objectives

Validation of our approach is related to the scenarios of interest for EBRC provider in Luxembourg and aims at:

  1. Comparing mono-objective vs. multi-objective optimization.
  2. Comparing multi-objective algorithms in terms of objectives satisfaction and scalability.

Research Questions

  1. What is the benefit of using multi-objective optimization compared to mono-objective one? How faster a good solution is found using a mono-objective?
  2. Which algorithm: CMOOP-epsilon Dominance, CMOOP without Epsilon Dominance and Random algorithm better satisfies the objectives while scaling to more complex cloud infrastructure model?

Experimental Setup

EBRC allocates to one of its customers a cluster composed of 5 physical machines to host its on-line stock trading website that is composed of the following components:

  1. A database to store items
  2. A load balancer
  3. A database for payment
  4. A database to manage users
  5. A web Front-end

Threats to Validity

Internal Validity:
  1. Number of generations and populations at the initial setup
  2. Stopping criterion
  3. Fitness functions models
External validity:
  1. Pertinence of the chosen optimization axis
  2. Selected resolution strategies

Conclusion

  1. Supervision framework as a supporting tool to reconfigure customer cloud infrastructure
  2. Effectiveness of search-based approaches to resolve a cloud multi-objective optimization problem
  3. Implementation using Kevoree as Models@run.time platform
  4. Effectiveness of the optimization at the level of architectural models: reasonable solutions in reasonable time, acceptable in the case of cloud dynamic reconfiguration

Future Work

  1. Detection of the absence of improvements as a stopping criterion
  2. Usage of Hyper-heuristics to improve the search performance
